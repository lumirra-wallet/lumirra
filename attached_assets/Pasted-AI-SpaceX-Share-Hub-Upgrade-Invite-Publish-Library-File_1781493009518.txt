Here's a complete breakdown of the architecture:

🔌 WebSockets — Not Used
There are no WebSockets in this project. No ws, socket.io, or native WebSocket references anywhere. Real-time feel is achieved differently (see below).

📡 "Real-Time" — How It Actually Works
HTTP Polling via React Query:

Dashboard data (purchases, transfers, summary) → polls every 15 seconds (refetchInterval: 15_000)
Price quotes → polls every 60 seconds (refetchInterval: 60 * 1000)
Simulated Live Price UI:

A setInterval runs client-side every 2 seconds, randomly fluctuating the displayed price for visual effect — it's cosmetic, not real server data.
🌐 No vercel.json
The project has no vercel.json. It runs as a standard Node.js server (Express), not a Vercel serverless deployment. The API server starts via node ./artifacts/api-server/dist/index.mjs.

🖥️ Express App Structure
Defined in artifacts/api-server/src/app.ts, exported as default app
Entry point: artifacts/api-server/src/index.ts — listens on a standard HTTP port
⚙️ Background Jobs
None. No Bull, Agenda, cron jobs, or job queues. Instead:

Prices — fetched on-demand from Yahoo Finance with in-memory caching inside the route handler
Emails — fire-and-forget async calls within the request lifecycle (no queue)
📦 Key Dependencies
Side	Packages
Server	express, mongoose, jsonwebtoken, nodemailer, pino
Client	@tanstack/react-query (polling), recharts (charts), framer-motion (animations), wouter (routing)
Bottom line: This is a traditional REST API + polling architecture. No WebSockets, no Vercel serverless, no background workers. The "live" feel on the dashboard is a combination of periodic HTTP polling and client-side animation tricks.
